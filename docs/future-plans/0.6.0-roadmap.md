# Opossum 0.6.0 --- Release Roadmap

> **Status:** Planning
> **Target release:** 0.6.0
> **Builds on:** 0.5.0 (durability and disaster recovery)
> **Theme:** Desktop production readiness --- GDPR/encryption + write throughput + code quality
> **Strategic context:** [Production Feasibility Analysis](../analysis/production-feasibility-analysis.md)

---

## Vision

Opossum 0.6.0 transforms the library from a learning/prototyping tool into a
**production-grade embedded event store for desktop and tablet applications**.

The [production feasibility analysis](../analysis/production-feasibility-analysis.md)
identified one viable niche: **single-user / small-team desktop apps** in regulated
domains (healthcare, legal, finance, quality inspection). For this niche:

- Throughput is irrelevant (humans generate <5 events/sec; Opossum does 55/sec)
- Zero infrastructure IS the killer feature (no database server, just .NET DLLs)
- Human-readable JSON files ARE the product differentiator (auditors can verify records)
- DCB fills a real gap (no other .NET library provides embedded DCB)

The two **missing capabilities** that block production use are:

1. **Encryption at rest** --- practitioners and advisors handle sensitive data
2. **GDPR data erasure** --- EU market requires "right to be forgotten"

These are the primary deliverables for 0.6.0. Throughput improvements and code quality
items are secondary but included since they were already designed and deferred from
previous releases.

---

## Release Structure: Three Phases

### Phase 1: GDPR Foundation (Soft Delete + Tombstones)

Low-effort, high-value. Enables basic data erasure without encryption.

### Phase 2: Encryption at Rest (Crypto Shredding)

Medium-effort, core feature. Enables the complete GDPR erasure story and protects PII.

### Phase 3: Throughput Optimizations + Code Quality

Previously deferred items from 0.4.0/0.5.0. Nice-to-have for 0.6.0 but not blocking.

## Phase 1: GDPR Foundation (Soft Delete + Tombstones)

> **Full design:** [docs/design/gdpr-and-encryption-design.md --- Soft Delete section](../design/gdpr-and-encryption-design.md#soft-delete-detailed-design)

### G1. Soft Delete (Tombstones)

**What:** Replace individual event files with tombstone markers that preserve the
position but remove all payload and PII data.

**API additions:**

```csharp
// On IEventStoreAdmin
Task SoftDeleteEventAsync(long position, string reason, CancellationToken ct = default);
Task SoftDeleteEventsAsync(Query query, string reason, CancellationToken ct = default);
```

**Tombstone format:** See [GDPR design doc](../design/gdpr-and-encryption-design.md#tombstone-format).

**Tasks:**
1. Create `TombstoneEvent : IEvent` record type in `Opossum.Core`
2. Add `SoftDeleteEventAsync` to `IEventStoreAdmin` interface
3. Implement in `FileSystemEventStore.Admin.cs`:
   - Acquire cross-process lock
   - Read event file to verify it exists and is not already a tombstone
   - If `WriteProtectEventFiles`, remove read-only attribute
   - Write tombstone JSON via temp-file + move pattern
   - If `WriteProtectEventFiles`, re-apply read-only attribute
4. Add `SoftDeleteEventsAsync` (resolves query to positions, calls single-delete in loop)
5. Add tombstone filtering to `ReadAsync` (skip events with `eventType == "$tombstone"`)
6. Add tombstone filtering to `ReadLastAsync`
7. Unit tests: single delete, query delete, double-delete idempotency, position preservation
8. Integration tests: append -> soft delete -> read (verify filtered), write-protection handling
9. Update CHANGELOG.md

**Estimated effort:** 2 sessions

### G2. ReadOption.IncludeTombstones

**What:** A new `ReadOption` value that includes tombstone events in query results
(for audit purposes).

**Tasks:**
1. Add `IncludeTombstones` to `ReadOption` enum
2. Modify tombstone filtering in `ReadAsync` to check for this option
3. Unit tests
4. Update CHANGELOG.md

**Estimated effort:** 1 session (can be done together with G1)

### G3. Projection Tombstone Awareness

**What:** Projections encountering a tombstone event skip it (no `Apply` call).
`ProjectionManager` and `ProjectionRebuilder` both need this.

**Tasks:**
1. Add tombstone check in `ProjectionManager.ProcessEventAsync` (skip if `$tombstone`)
2. Add tombstone check in `ProjectionRebuilder` replay loop
3. Unit tests: projection receiving tombstone does not call Apply
4. Integration tests: append -> project -> soft delete -> rebuild -> verify state unchanged
5. Update CHANGELOG.md

**Estimated effort:** 1 session

## Phase 2: Encryption at Rest (Crypto Shredding)

> **Full design:** [docs/design/gdpr-and-encryption-design.md --- Crypto Shredding section](../design/gdpr-and-encryption-design.md#crypto-shredding-detailed-design)
> **GDPR reference:** [Oskar Dudycz - GDPR in Event-Driven Architecture](https://event-driven.io/en/gdpr_in_event_driven_architecture/)

### E1. [PersonalData] Attribute + Encryption Config

**What:** A `[PersonalData]` attribute for marking PII fields on event types, plus
`EncryptionOptions` configuration.

**Tasks:**
1. Create `[PersonalData]` attribute in `Opossum.Core` with optional `SubjectIdField` property
2. Create `EncryptionOptions` class in `Opossum.Configuration`:
   - `byte[] MasterKey` (or `DeriveFromPassword(string password, byte[] salt)`)
   - `Func<DomainEvent, string?> SubjectIdResolver`
   - `string RedactedValue` (default: `"[REDACTED]"`)
3. Add `Encryption` property to `OpossumOptions` (lazy-initialized `EncryptionOptions`)
4. Add `EncryptionOptions.Enable(Action<EncryptionOptions>)` fluent API
5. Add validation in `OpossumOptionsValidator`: if encryption enabled, MasterKey must be set
6. Unit tests: attribute discovery via reflection, options validation
7. Update CHANGELOG.md

**Estimated effort:** 1-2 sessions

### E2. Subject Key Store (File-Based)

**What:** A file-based key management system that stores one AES-256 key per data
subject, encrypted with the master key.

**Directory structure:**
```
MyStore/Keys/subjects/{subjectId}.key    # AES-256 key encrypted with master key
MyStore/Keys/master.meta                  # Master key metadata (salt, algorithm, version)
```

**Tasks:**
1. Create `SubjectKeyStore` internal class in `Opossum.Storage.FileSystem`
2. Implement `GetOrCreateKeyAsync(string subjectId)`:
   - If `{subjectId}.key` exists, read and decrypt with master key
   - If not, generate random AES-256 key, encrypt with master key, write to file
3. Implement `DeleteKeyAsync(string subjectId)`:
   - Delete `{subjectId}.key` file
   - Return true/false for success
4. Implement `GetSubjectIdsAsync()`:
   - List all `.key` files in the `subjects/` directory
5. Implement `RotateMasterKeyAsync(byte[] oldMasterKey, byte[] newMasterKey)`:
   - Read each `.key` file, decrypt with old key, re-encrypt with new key, atomic move
6. Use AES-256-GCM for key wrapping (encrypt subject key bytes with master key)
7. Key file format: JSON with `{ "encryptedKey": "base64", "nonce": "base64", "tag": "base64", "createdAt": "..." }`
8. Use temp-file + atomic move for all writes (consistent with Opossum patterns)
9. Respect `FlushEventsImmediately` for key file writes (durability guarantee)
10. Update `StorageInitializer` to create `Keys/subjects/` directory
11. Unit tests: create, read, delete, rotate, concurrent access
12. Integration tests: key persistence across restarts

**Estimated effort:** 2-3 sessions

### E3. Encrypting Serializer

**What:** A serializer wrapper that encrypts `[PersonalData]` fields on write and
decrypts on read, using the subject's encryption key.

**Tasks:**
1. Create `EncryptingJsonEventSerializer` that wraps `JsonEventSerializer`
2. **Write path** (`Serialize`):
   a. Serialize event to `JsonDocument` (or `JsonNode`) first
   b. Scan for `[PersonalData]` fields via reflection (cache results per type)
   c. Resolve subject ID from the event (via attribute or resolver delegate)
   d. Get/create subject key from `SubjectKeyStore`
   e. For each PII field: encrypt value with AES-256-GCM, replace with `{ "$encrypted": "...", "$nonce": "...", "$tag": "..." }`
   f. Serialize modified JSON to string
3. **Read path** (`Deserialize`):
   a. Parse JSON, detect `$encrypted` markers
   b. Resolve subject ID from the event
   c. Attempt to get subject key from `SubjectKeyStore`
   d. If key exists: decrypt each marked field, restore plaintext
   e. If key deleted: replace with `RedactedValue` (configurable)
   f. Deserialize to `SequencedEvent`
4. Cache reflection results (`[PersonalData]` field discovery) per event type
5. Handle mixed stores: events without `$encrypted` markers are treated as plaintext
   (backward compatible with pre-encryption events)
6. Register `EncryptingJsonEventSerializer` in DI when encryption is enabled
7. Unit tests: encrypt roundtrip, key-not-found handling, mixed encrypted/plain events,
   performance impact measurement
8. Integration tests: full append -> read cycle with encryption, cross-restart persistence

**Estimated effort:** 3-4 sessions

### E4. GDPR Erasure API

**What:** User-facing API for exercising the right to erasure via crypto shredding.

**API additions:**

```csharp
// On IEventStoreAdmin
Task<bool> EraseSubjectKeyAsync(string subjectId, CancellationToken ct = default);
Task<IReadOnlyList<string>> GetSubjectIdsAsync(CancellationToken ct = default);
```

**Tasks:**
1. Add `EraseSubjectKeyAsync` to `IEventStoreAdmin` --- delegates to `SubjectKeyStore.DeleteKeyAsync`
2. Add `GetSubjectIdsAsync` to `IEventStoreAdmin` --- delegates to `SubjectKeyStore.GetSubjectIdsAsync`
3. After key deletion, affected events are automatically redacted on next read (no event file modification needed)
4. Document: recommend rebuilding affected projections after erasure
5. Unit tests: erase -> read returns redacted, erase nonexistent returns false
6. Integration tests: full GDPR flow (append with PII -> erase -> read -> verify redacted)
7. Update CHANGELOG.md

**Estimated effort:** 1 session

### E5. Key Rotation API

**What:** Rotate the master key without touching event files.

**API additions:**

```csharp
// On IEventStoreAdmin
Task RotateMasterKeyAsync(byte[] newMasterKey, CancellationToken ct = default);
```

**Tasks:**
1. Add `RotateMasterKeyAsync` to `IEventStoreAdmin`
2. Delegates to `SubjectKeyStore.RotateMasterKeyAsync`
3. Re-encrypts all subject key files with the new master key
4. Atomic: if any key file fails, roll back all changes
5. Unit tests: rotation, verify old events still readable with new master key
6. Integration tests: rotate -> append -> read -> verify

**Estimated effort:** 1 session

## Phase 3: Throughput Optimizations + Code Quality

> **Full design:** [docs/design/throughput-optimization-plan.md](../design/throughput-optimization-plan.md)
> **Research:** [docs/analysis/throughput-research-and-improvement-paths.md](../analysis/throughput-research-and-improvement-paths.md)

These items were originally scoped for 0.4.0, deferred to 0.5.0, and deferred again
to 0.6.0. They are not blocking for the desktop niche (throughput is irrelevant at
human speed) but improve the overall quality of the library.

### T1. Option E --- Implicit Ledger

> **Full design:** [Throughput plan --- Option E](../design/throughput-optimization-plan.md#option-e-implicit-ledger-eliminate-the-second-fsync)

**What:** Remove the `.ledger` file. Derive position from directory scan on startup.
Saves one fsync per append (~2.5ms).

**Breaking:** Yes (`.ledger` file disappears). Migration: transparent (scan on startup).

**Tasks:** See [throughput plan, 10 tasks](../design/throughput-optimization-plan.md#implementation-tasks).

**Estimated effort:** 2-3 sessions

### T2. Option A --- Append-Only Index Files

> **Full design:** [Throughput plan --- Option A](../design/throughput-optimization-plan.md#option-a-append-only-index-files)

**What:** Replace JSON array index files with newline-delimited append-only files.
Eliminates read-modify-write cycles.

**Breaking:** Yes (`.json` -> `.idx` format). Migration: automatic on startup.

**Tasks:** See [throughput plan, 11 tasks](../design/throughput-optimization-plan.md#implementation-tasks-1).

**Estimated effort:** 2 sessions

### T3. Option B --- In-Memory Index Cache

> **Full design:** [Throughput plan --- Option B](../design/throughput-optimization-plan.md#option-b-in-memory-index-cache)

**What:** Cache all indices in memory. Serve reads from cache. Refresh on cross-process
lock acquisition.

**Breaking:** No.

**Dependency:** T2 (Option A) should be implemented first.

**Tasks:** See [throughput plan, 9 tasks](../design/throughput-optimization-plan.md#implementation-tasks-2).

**Estimated effort:** 3-4 sessions

### T4. Cache Compiled Delegate in ReflectionMessageHandler

**What:** Cache the compiled handler delegate after first invocation instead of
reflecting on every call.

**Breaking:** No.

**Design:** See [0.5.0 roadmap](0.5.0-roadmap.md).

**Estimated effort:** 1-2 sessions

### T5. ProjectionTagIndex._tagLocks Unbounded

**What:** The `ConcurrentDictionary<string, SemaphoreSlim>` in `ProjectionTagIndex`
grows without bound. Add LRU eviction or use a fixed pool.

**Breaking:** No.

**Estimated effort:** 1 session

### T6. Parallel Read Threshold Magic Numbers

**What:** Replace hardcoded parallel read thresholds with configurable options.

**Breaking:** No.

**Estimated effort:** 1 session

## Breaking Changes Summary

| Item | What breaks | Migration | User impact |
|------|-------------|-----------|-------------|
| T1 | `.ledger` file disappears | Transparent (scan on startup) | None for users; stores just work |
| T2 | Index format `.json` -> `.idx` | Automatic on startup | None for users; auto-migrated |
| G1 | New `$tombstone` event type | N/A (new capability) | None; tombstones filtered by default |
| E1-E5 | New `Keys/` directory | Created automatically | None; encryption is opt-in |

**Note:** Ship all breaking format changes (T1, T2) while the installed base is
still near zero to avoid migration burden for future users.
